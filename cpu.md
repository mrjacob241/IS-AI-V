# cpu.v

`cpu.v` is the top-level RV32I single-cycle CPU. It connects the existing
building blocks in this directory:

- `control.v`
- `imm_gen.v`
- `regfile.v`
- `alu.v`
- `ram.v`
- `rom_dump.v` (auto-generated by `parse_rom.py`)

The CPU fetches one 32-bit instruction per cycle, decodes it, executes it, and
updates the program counter on the rising edge of `clk`.

## Interface

```verilog
module cpu (
    input  wire        clk,
    input  wire        rst,

    input  wire        dbg_ram_we,
    input  wire [31:0] dbg_ram_addr,
    input  wire [31:0] dbg_ram_wd,
    input  wire        dbg_rf_we,
    input  wire [4:0]  dbg_rf_rd,
    input  wire [31:0] dbg_rf_wd,

    output wire [31:0] dbg_pc,
    output wire [31:0] dbg_x1,
    output wire [31:0] dbg_x2,
    output wire [31:0] dbg_x3,
    output wire [31:0] dbg_x4,
    output wire [31:0] dbg_x10,
    output wire [31:0] dbg_instr,
    output wire        dbg_rom_loading,

    output wire [23:0] video_rgb,
    output wire        video_hsync,
    output wire        video_vsync,
    output wire        video_de,
    output wire [9:0]  video_x,
    output wire [8:0]  video_y,
    output wire        video_frame_done
);
```

## Normal CPU Ports

| Port | Direction | Meaning |
|------|-----------|---------|
| `clk` | input | Main clock. PC, register writes, and RAM writes update on the rising edge. |
| `rst` | input | Synchronous reset for the PC. When high at a rising edge, `pc <= 0`. |

## Debug and Initialization Ports

The `dbg_*` input ports exist for Verilator testbenches. They allow tests to
load RAM and initialize registers without reaching into generated simulator
internals.

| Port | Direction | Meaning |
|------|-----------|---------|
| `dbg_ram_we` | input | Write enable for debug RAM load. |
| `dbg_ram_addr` | input | Byte address for debug RAM write. |
| `dbg_ram_wd` | input | 32-bit instruction/data word for debug RAM write. |
| `dbg_rf_we` | input | Write enable for debug register-file initialization. |
| `dbg_rf_rd` | input | Register index to initialize. |
| `dbg_rf_wd` | input | Register value to initialize. |

When `dbg_ram_we` is high, the CPU writes `dbg_ram_wd` to RAM at
`dbg_ram_addr` as a word store. When `dbg_rf_we` is high, the CPU writes
`dbg_rf_wd` to register `dbg_rf_rd`.

Debug writes have the highest priority and work regardless of `rst` state.
Normal instruction writes (`reg_write`, `mem_write`) are suppressed while
`rst=1` so that the instruction fetched at PC=0 during reset cannot corrupt
registers or memory. Only debug-port writes pass through during reset.

The testbench holds `rst` high while loading RAM and initializing registers
so the PC does not advance during that phase.

## Debug Output Ports

| Port | Meaning |
|------|---------|
| `dbg_pc` | Current program counter. |
| `dbg_x1` | Register x1, return address `ra`. |
| `dbg_x2` | Register x2, stack pointer `sp`. |
| `dbg_x3` | Register x3. |
| `dbg_x4` | Register x4. |
| `dbg_x10` | Register x10, ABI name `a0`. |
| `dbg_instr` | Current instruction word fetched from RAM at `dbg_pc`. |
| `dbg_rom_loading` | High while `rst=1` and ROM words remain to be written. Testbenches poll this instead of using a fixed cycle count. |
| `video_*` | Virtual HDMI-style scanout from `gpu.v`: RGB, sync/data-enable, pixel coordinates, and frame marker. |

`dbg_instr` is used by the integration test to stop when it sees
`32'h00000073`, the RV32I `ecall` instruction.

## Instruction Decode

The CPU fetches `instr` from RAM and splits it into standard RV32I fields:

```verilog
opcode = instr[6:0];
rd     = instr[11:7];
funct3 = instr[14:12];
rs1    = instr[19:15];
rs2    = instr[24:20];
funct7 = instr[31:25];
```

Those fields feed `control.v`, while the full instruction feeds `imm_gen.v`.

## Datapath

### Register File

`regfile.v` provides two combinational reads:

- `rs1_data = x[rs1]`
- `rs2_data = x[rs2]`

The write address and write data normally come from the decoded instruction and
writeback mux. During debug initialization, `dbg_rf_*` overrides the normal
write path.

### ALU Input A

```verilog
alu_a = lui ? 0 : (pc_to_alu ? pc : rs1_data);
```

Most instructions use `rs1_data`. AUIPC uses `pc`, selected by the control
signal `pc_to_alu`. LUI forces input A to zero because its apparent `rs1`
bits are part of the upper immediate, not a register operand.

### ALU Input B

```verilog
alu_b = alu_src ? imm : rs2_data;
```

Register-register instructions use `rs2_data`. Immediate, load/store address,
JALR address, LUI, and AUIPC paths use `imm`.

### Writeback

```verilog
wb_data =
    (mem_to_reg == 2'b01) ? ram_rd :
    (mem_to_reg == 2'b10) ? pc_plus_4 :
                            alu_y;
```

Writeback sources:

| `mem_to_reg` | Source | Used by |
|--------------|--------|---------|
| `00` | ALU result | ALU ops, LUI, AUIPC |
| `01` | RAM read data | Loads |
| `10` | `PC + 4` | JAL, JALR |

## Program Counter

The PC is a 32-bit register. It resets to zero and otherwise updates to
`next_pc` on each rising edge.

```verilog
if (rst)
    pc <= 32'b0;
else
    pc <= next_pc;
```

The next-PC priority is:

1. JALR target
2. JAL target
3. Taken branch target
4. Sequential `PC + 4`

In Verilog:

```verilog
next_pc =
    jalr         ? jalr_target :
    jump         ? jal_target :
    branch_taken ? branch_target :
                   pc_plus_4;
```

Targets:

```verilog
pc_plus_4     = pc + 4;
branch_target = pc + imm;
jal_target    = pc + imm;
jalr_target   = (rs1_data + imm) & 32'hFFFF_FFFE;
```

JALR clears bit 0 as required by the RISC-V ISA.

## Branch Logic

Branches reuse the ALU result.

```verilog
branch_predicate = (alu_op == ALU_SUB) ? alu_zero : alu_y[0];
branch_taken = branch & (branch_inv ^ branch_predicate);
```

BEQ and BNE use subtraction:

- BEQ takes the branch when `zero == 1`.
- BNE sets `branch_inv`, so it takes the branch when `zero == 0`.

BLT, BGE, BLTU, and BGEU use SLT or SLTU:

- BLT and BLTU take the branch when `alu_y[0] == 1`.
- BGE and BGEU set `branch_inv`, so they take the branch when `alu_y[0] == 0`.

## Memory

`ram.v` is a unified byte-addressable memory used for both instruction fetch and
data access.

- Instruction fetch uses `iaddr = pc`.
- Data access normally uses `daddr = alu_y`.
- Stores write `rs2_data`.

GPU MMIO is decoded before normal RAM access:

| Address range | Target |
|---------------|--------|
| `0x00000000` - `0x0001FFFF` | `ram.v` |
| `0x00020000` - `0x00032BFF` | `gpu.v` framebuffer |
| `0x00040000` - `0x000400FF` | `gpu.v` registers |

Stores to GPU ranges update the GPU instead of RAM. Loads from GPU ranges return
GPU framebuffer/register data through the normal writeback path.
- Loads read `ram_rd`.
- `mem_size` selects byte, halfword, or word access.
- `mem_unsigned` selects sign-extension or zero-extension for byte/halfword
  loads.

The RAM write path uses a three-way priority mux:

1. `dbg_ram_we` — debug word write (highest priority, works during reset)
2. `rom_loading` — ROM loader streaming words during reset
3. `mem_write` — normal store instruction (suppressed during reset)

`rom_loading` is active while `rst=1` and the ROM counter has not yet
exhausted all words in `rom_dump`. Each cycle during reset one word is
written to RAM at its byte address supplied by `rom_addr` — which is the
actual ELF byte address for that word, not just `rom_idx * 4`. This allows
non-contiguous sections (e.g. `.text` at 0x0000 and `.rodata` at 0x03D0) to
be loaded correctly.

## Supported Instruction Classes

Support comes from the connected `control`, `imm_gen`, `alu`, and `ram`
modules. The integrated CPU path supports:

- R-type ALU operations
- I-type ALU operations
- Loads and stores
- Conditional branches
- JAL and JALR
- LUI and AUIPC
- SYSTEM instructions as halt markers for the testbench

`ecall` is not handled as a trap inside hardware. The testbench detects
`dbg_instr == 32'h00000073` and stops simulation.

## ROM Loader

`cpu.v` instantiates `rom_dump` as a sub-module and streams its words into
RAM automatically during reset. `rom_dump.v` is generated by `parse_rom.py`:

```bash
python3 parse_rom.py codes/instr/<name>_instr.txt   # writes rom_dump.v
```

`rom_dump` is a purely combinational module: given a word index it returns the
corresponding 32-bit data word, its actual byte address (`rom_addr`), and a
`valid` flag. Inside `cpu.v` a counter register `rom_idx` increments on each
rising edge while `rst=1` and `rom_valid=1`. The `rom_addr` output carries
the ELF byte address for that word, so non-sequential sections (`.text` then
`.rodata`) are placed at the correct RAM locations.

Once `rom_idx >= NWORDS`, `rom_valid` goes low and the counter stops.
`dbg_rom_loading` (`= rst & rom_valid`) is exported so testbenches can poll
it rather than using a hardcoded cycle count.

Testbench reset protocol:

```cpp
top.rst = 1;
top.eval();
while (top.dbg_rom_loading) tick(top);   // wait for all words to land in RAM

top.dbg_rf_we = 1; top.dbg_rf_rd = 2; top.dbg_rf_wd = 0x0001FFF0;
tick(top);                               // initialize sp (still during rst)
top.dbg_rf_we = 0;

top.rst = 0;                             // release reset, CPU runs from PC=0
```

This works for programs of any size without adjustment.

## Related C++ Files

### `tb_cpu.cpp` — integration test

Loads short inline programs for each instruction class and checks results.

| Program | Coverage |
|---------|----------|
| A | ADDI, ADD, SUB, basic register writeback |
| B | SW, LW, unified RAM data path |
| C | BEQ not taken |
| D | BEQ taken |
| E | JAL, JALR, `PC + 4` link writeback |
| F | LUI plus ADDI constant construction |
| G | Compiled binary from `rom_dump.v`, `sp=0x0001FFF0`, assert `a0==5` at ecall |

```bash
./run.sh cpu tb_cpu.cpp
```

### `tb_debug.cpp` — cycle-by-cycle tracer

Loads `rom_dump.v` program via the reset ROM loader, then traces execution
one instruction at a time. Each line shows the cycle number, PC, raw
instruction hex, decoded assembly, and the registers that changed that cycle
(before → after). Stops at `ecall` and prints the final value of `a0`.

Register values are read directly from `cpu->rootp->cpu__DOT__rf__DOT__regs[i]`,
which Verilator exposes without any `--public` flag.

```bash
./run.sh cpu tb_debug.cpp
```

Example output:

```
[  0]  0x0000  FE010113  addi   sp, sp, -32     sp: 0x0001FFF0→0x0001FFD0
[  1]  0x0004  00812E23  sw     s0, 28(sp)       (no change)
...
[ 13]  0x0034  00000073  ecall                   (no change)
----
HALT — ecall at PC=0x0034   a0 = 5 (0x00000005)
```

### `parse_rom.py` — objdump → Verilog ROM

Converts a RISC-V objdump disassembly into `rom_dump.v`. Understands two
input formats emitted by `build_riscv.sh`:

- **`objdump -d`** lines (`addr:  8hexdigits  disasm`) for `.text` — the 8-hex
  value is already the little-endian 32-bit word.
- **`objdump -s`** lines (` addr 8bytes ...`) for `.rodata` and `.srodata` —
  each 4-byte group is shown in address order, so `struct.unpack('<I', ...)` is
  applied to get the correct little-endian word for RAM.

Both formats can appear in the same file; the parser deduplicates by address
(`.text` words from `-d` take priority, `.rodata` words come only from `-s`).

```bash
python3 parse_rom.py <objdump.txt> [rom_dump.v]
```

The generated `rom_dump` module has three outputs: `data` (32-bit word),
`addr` (actual ELF byte address), and `valid`.
