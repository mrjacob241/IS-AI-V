# CPU Program Tests

Compiled RV32I test programs for the single-cycle CPU.
Each program is a bare-metal C++ binary linked at `0x00000000`.

For the GPU framebuffer, virtual screen capture, and Pong rendering pipeline,
see `GPU_RENDERING.md`.

---

## Quick start

```bash
# Compile + objdump (once per program)
./codes/build_riscv.sh <name>

# Verify on host with qemu (exit code = a0 & 0xFF)
./codes/build_and_run.sh <name>

# Parse ROM + Verilator compile + run on CPU
./test.sh <name> <expected_a0>

# Trace execution cycle-by-cycle (debug)
python3 parse_rom.py codes/instr/<name>_instr.txt
./run.sh cpu tb_debug.cpp

# Interactive I/O programs
python3 parse_rom.py codes/instr/<name>_instr.txt
./run.sh cpu tb_io.cpp

# Live GPU screen programs
python3 ppm_live_viewer.py screen.ppm
python3 parse_rom.py codes/instr/pong_live_instr.txt
./run.sh cpu tb_live_screen.cpp

# Player-controlled Pong
./codes/build_riscv.sh pong_test
python3 parse_rom.py codes/instr/pong_test_instr.txt
./run.sh cpu tb_pong_test.cpp
```

## Pipeline internals

### ROM loading

`cpu.v` instantiates `rom_dump` (generated by `parse_rom.py`) and a counter
`rom_idx` that increments while `rst=1` and `rom_valid=1`. Each cycle one
word is written into RAM. The word's destination address comes from the ROM's
`rom_addr` output тАФ the actual ELF byte address тАФ so non-contiguous sections
(`.text` at 0x0000, `.rodata` at 0x03D0) land at the correct locations.

`cpu.v` exports `dbg_rom_loading` (`= rst & rom_valid`). Testbenches poll
it to know when loading is complete, replacing the old fixed-cycle approach.

### Reset protocol

```cpp
top.rst = 1;
top.eval();
while (top.dbg_rom_loading) tick(top);   // works for any program size

top.dbg_rf_we = 1; top.dbg_rf_rd = 2; top.dbg_rf_wd = 0x0001FFF0;
tick(top);                               // sp = top of stack (still rst=1)
top.dbg_rf_we = 0;

top.rst = 0; top.eval();                 // CPU starts at PC=0
```

### build_riscv.sh output format

The `_instr.txt` file contains two stacked objdump outputs:

1. `objdump -d` тАФ human-readable `.text` disassembly (8-hex-digit instruction words)
2. `objdump -s -j .text -j .rodata -j .srodata` тАФ raw section bytes (4-byte groups in address order)

`parse_rom.py` reads both. It uses `int(hex, 16)` for `-d` lines and
`struct.unpack('<I', bytes.fromhex(grp))` for `-s` lines to get the correct
little-endian word. `.text` words are deduplicated (first seen wins, so `-d`
takes priority).

### .rodata rules

- String literals and `static const` data go into `.rodata` / `.srodata`.
- The linker script merges `.srodata*` into `.rodata` and pads to `ALIGN(4)`,
  ensuring every word in the section is a complete 4-byte group.
- Programs that use string literals need `tb_io.cpp` (not `tb_program.cpp`)
  since they are interactive.
- Avoid `*` and `/` operators in C: the build uses `-march=rv32im` but the CPU
  only implements RV32I. Use `(x<<3)+(x<<1)` for ├Ч10, subtraction loops for division.

### Memory layout

```
0x00000000 тАУ 0x0000xxxx   .text    (loaded by ROM)
0x0000xxxx тАУ 0x0000yyyy   .rodata  (loaded by ROM, may have a gap before stack)
0x0001FFD0 тАУ 0x0001FFF0   stack    (grows down from 0x1FFF0)
```

### Ecall conventions for `tb_io.cpp`

| a7 | Action |
|----|--------|
| 1 | Print signed integer in a0 |
| 11 | Print character (a0 & 0xFF) |
| 12 | Legacy read one character тЖТ a0 (interactive programs should prefer `IO_SDK`) |
| 93 | Exit (break simulation loop) |

### `IO_SDK` keyboard input

Programs can include `../../IO_SDK.h` to read keyboard input from `io.v`.
`codes/build_riscv.sh` automatically links `../IO_SDK.cpp`.

`io.v` lives at `0x00050000`:

| Address | Meaning |
|---------|---------|
| `0x00050000` | Key-state bitmask (`IO_KEY_W`, `IO_KEY_S`, `IO_KEY_Q`) |
| `0x00050004` | Current typed character |
| `0x00050008` | Typed-character sequence |
| `0x0005000C` | Typed-character acknowledge |

`Keyboard.KeysStream(...)` uses the sequence/acknowledge pair so fast typed
input is queued instead of overwritten.

## Selected program notes

### `t_rodata` тАФ .rodata loading unit test

Source: `codes/src/t_rodata.cpp`
Reads the string literal `"Hello"` byte-by-byte and returns the sum of its
ASCII values: H(72)+e(101)+l(108)+l(108)+o(111) = **500**.

This is the focused unit test for the `.rodata` bug: string bytes in `.rodata`
were previously dropped because `objdump -D` emits 16-bit "insns" for them,
which the old 8-hex-digit regex silently skipped. The fix is the `-s` format
parsed with `struct.unpack('<I', ...)`.

### `calc` тАФ interactive two-number adder

Source: `codes/src/calc.cpp`
Prompts for two integers, prints their sum, repeats. Type `q` at either prompt
to quit.

```bash
python3 parse_rom.py codes/instr/calc_instr.txt
./run.sh cpu tb_io.cpp
```

Key implementation constraints (no RV32M):
- **Input**: uses `IO_SDK` `Keyboard.KeysStream(...)`; `tb_io.cpp` feeds `io.v`
  from a nonblocking host input thread.
- **`read_num`**: parses digits with `(num<<3)+(num<<1)+digit` (avoids `mul`)
- **`print_int`**: uses a stack-local powers-of-10 table + repeated subtraction (avoids `div`)
- String prompts live in `.rodata` and are loaded at reset along with `.text`

## Adding a new test

1. Write `codes/src/<name>.cpp` тАФ entry must be `extern "C" void _start()`.
   Forward-declare all helpers; define `_start` first so it lands at address 0x0.
2. End with `ecall` using a7=93 (exit) if using `tb_program.cpp`, or a7=93 to break the loop in `tb_io.cpp`.
3. Verify on host: `./codes/build_and_run.sh <name>` тАФ exit code must equal expected a0 (if тЙд 127).
4. Compile for CPU: `./codes/build_riscv.sh <name>`
5. Run on CPU: `./test.sh <name> <expected_a0>`
6. On failure, trace: `python3 parse_rom.py codes/instr/<name>_instr.txt && ./run.sh cpu tb_debug.cpp`

**Pitfalls:**
- If a helper is defined before `_start`, GCC places it at 0x0000 and the CPU executes it on boot (infinite loop or garbage). Always forward-declare helpers and define `_start` first.
- Keep expected a0 тЙд 127 for `build_and_run.sh` verification (qemu exit code truncates to 8 bits).
- Avoid `*` and `/` тАФ they emit `mul`/`div` (RV32M), which the CPU doesn't implement.
- String literals require `tb_io.cpp` if the program also does I/O; otherwise `tb_program.cpp` is fine since `.rodata` is now loaded correctly.
